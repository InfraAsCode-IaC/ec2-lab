# Terraform EC2 AWS

Terraform project designed to deploy an EC2 instance on AWS with a modular, secure, and reproducible architecture.

## Overview

This implementation separates infrastructure into three main modules:

- `modules/networking`: creates the VPC, public and private subnets, Internet gateway, and public route table.
- `modules/security`: defines the security group for the EC2 instance, enabling controlled SSH access and web traffic.
- `modules/compute`: deploys the EC2 instance into a public subnet, assigns an Elastic IP, and configures the encrypted root volume.

The environment layer (`environments/dev`, `environments/prod`) instantiates these modules and provides environment-specific parameters.

## Requiriments 💡

Before running Terraform, ensure the following are in place:

- Create a dedicated IAM user for Terraform operations.
- Configure the AWS CLI profile with the appropriate credentials.
- Create an S3 bucket to store remote state.
- Enable versioning on the S3 bucket.
- Enable AES-256 encryption on the S3 bucket.
- Create a DynamoDB table for state locking (`tfstate`).
- Install Terraform >= 1.7.0.
- Install AWS CLI.
- Have a valid SSH key available to access the EC2 instance.

## Module behavior

### 1. `networking` module

This module provides the networking foundation for the EC2 instance:

- `aws_vpc.main`: primary VPC with DNS support enabled.
- `aws_internet_gateway.main`: Internet access for the VPC.
- `aws_subnet.public`: public subnet with automatic public IP assignment.
- `aws_subnet.private`: private subnet for future expansion.
- `aws_route_table.public` and `aws_route_table_association.public`: default route to the Internet gateway.

Key outputs consumed by other modules:

- `public_subnet_id`: subnet where the EC2 instance is launched.
- `vpc_id`: VPC used by the security module.

### 2. `security` module

This module encapsulates the network access policy for the instance:

- Defines an `aws_security_group` that permits SSH only from authorized IP addresses (`allowed_ssh_ips`).
- Allows HTTP (80) and HTTPS (443) traffic from the internet.
- Keeps outbound access open to allow updates and controlled outbound connectivity from the instance.

Key output consumed by the compute module:

- `security_group_id`: identifier of the security group applied to the EC2 instance.

### 3. `compute` module

This module provisions the EC2 instance with the following design decisions:

- Uses the latest Amazon Linux 2023 AMI.
- Deploys the instance into the provided `public_subnet_id`.
- Associates the security group generated by the `security` module.
- Creates an EC2 key pair (`aws_key_pair`) for SSH access.
- Attaches an encrypted `gp3` EBS root volume with `delete_on_termination = true`.
- Enforces IMDSv2 (`http_tokens = "required"`) to harden metadata access.
- Allocates an Elastic IP to ensure stable public connectivity.

## Environment strategy

Each environment contains its own Terraform configuration and backend:

- `environments/dev`: development configuration.
- `environments/prod`: production configuration.

The `dev` environment uses an S3 backend with DynamoDB state locking. This ensures shared remote state and consistency for collaborative teams.

## Backend configuration

The S3 backend in `environments/dev/backend.tf` is configured as an example:

- bucket: `iac.tfstate.dev`
- key: `dev/ec2/terraform.tfstate`
- region: `us-east-1`
- encrypt: `true`
- dynamodb_table: `iac-state-lock`

> Adjust these values to your environment and naming conventions before initializing Terraform.

## Recommended workflow

1. Configure environment-specific variables in `environments/<env>/terraform.tfvars`.
2. Run `terraform init` from the environment directory.
3. Run `terraform plan` to validate the proposed changes.
4. Run `terraform apply` to deploy the infrastructure.

##
